# 项目1 复述练习卡片（5题）

> 练法：先不看这个文件，对着手机录音讲。卡壳了再看。
> 目标：每题30秒版能脱口而出，2分钟版讲得流畅不卡壳。

---

## 题1：介绍一下这个项目

### 30秒版
这是一个 AI 学术研究助手，帮科研人员检索和管理论文。核心是三条路径：知识问答走 RAG，工具调用走 Agent，闲聊直接回复。检索层用 BM25+向量的混合检索加 CrossEncoder 精排，知识问答用 CRAG 三级降级防幻觉，工具层接了 ArXiv、Semantic Scholar、笔记管理三个 MCP Server，用 Plan-and-Execute 加人工确认保证安全。整体用 LangGraph 编排。

### 2分钟版
在30秒版基础上展开三个点：

**检索层为什么这么设计**：BM25 擅长精确关键词但没有语义理解，向量检索有语义理解但容易丢精确词汇，两者 RRF 融合互补。粗排拿20条候选，CrossEncoder 精排到 top-3，评测显示比单路检索 MRR 高了 30-40 个百分点。

**CRAG 解决什么问题**：传统 RAG 不管检索质量好坏都直接生成，容易幻觉。CRAG 先让 LLM 对每篇文档做三元评分，relevant 直接生成，ambiguous 改写 query 重试最多3次，全部 irrelevant 降级到 Web Search，最后兜底是 LLM 自身知识但会标注来源。

**Agent 怎么工作**：意图分类为 task 后，AgentPlanner 根据9个可用工具生成 JSON 执行计划，Human-in-the-Loop 让用户确认，ToolExecutor 逐步执行，最后 synthesis 节点综合结果生成回答。

### 记忆锚点
三条线：入口线（改写→分类→三选一）、知识线（混合检索→精排→CRAG→生成）、工具线（规划→确认→执行→综合）

## 题2：这个项目的难点或挑战是什么

### 30秒版
最大的挑战是保证检索质量和回答忠实度。学术论文专业性强，单一检索方式效果差，所以设计了 Hybrid+CrossEncoder 两阶段检索。检索到了还不够，文档不相关直接生成会幻觉，所以加了 CRAG 三级降级做质量把关。另外工具调用的安全性也是挑战，用 Human-in-the-Loop 让用户确认才执行。

### 2分钟版
三个难点分别展开：

**难点一：检索质量**。BM25 搜"文档检索"搜不到"information retrieval"，向量检索把"BERT-Large"和"GPT-2"搞混。解决：两路并行 + RRF 融合 + CrossEncoder 精排。4组对比实验验证，Hybrid+Reranker 最好，测试集10个用例 MRR=1.0。样本量不大但趋势明显。

**难点二：回答忠实度**。检索到文档不代表答案对，传统 RAG 硬生成容易幻觉。解决：CRAG 评分+三级降级+生成 Prompt 里严格限制"只用上下文信息"，temperature=0.1。系统从"有时候对"变成"要么对，要么诚实说不知道"。

**难点三：工具调用安全**。LLM 规划的工具步骤可能有幻觉（调不存在的工具、参数错）。解决：校验 tool 是否在 available_tools 里，限制最多3步，Human-in-the-Loop 确认，JSON 解析失败返回空计划而非崩溃。

### 记忆锚点
三个难点：检索质量（Hybrid+精排）、回答忠实度（CRAG+忠实度约束）、工具安全（校验+人工确认）

## 题3：CRAG 是怎么做的

### 30秒版
CRAG 解决检索结果质量不稳定的问题。精排拿到 top-3 文档后，LLM 逐篇做三元评分：relevant、ambiguous、irrelevant。超过50%是 relevant 就直接生成；ambiguous 就改写 query 重新检索，最多重试3次；全部 irrelevant 或重试超限，降级到 DuckDuckGo Web Search。这样保证每条路都有兜底，不会用差的检索结果硬生成幻觉。

### 2分钟版
在30秒版基础上回答三个 why：

**为什么用 LLM 评分而不是用相似度分数？** 相似度高不代表真相关——向量空间里"注意力机制"和"注意力缺陷"可能很近，但答非所问。LLM 能理解语义层面的相关性，判断这个文档到底能不能回答用户的问题。

**改写怎么做的？** 取前2篇文档的中间段落（跳过前30字避免页眉，取300字）作为上下文，辅助 LLM 生成更精确的检索词。temperature=0.3 给一点随机性避免改写跟原 query 太像。

**50%阈值怎么来的？** 调测试集调出来的。太高容易触发过多不必要重试，太低会放过质量差的文档。3篇文档的情况下50%意味着至少2篇 relevant 才直接生成。

### 记忆锚点
三级降级：relevant→直接生成，ambiguous→改写重试（最多3次），irrelevant→Web Search

## 题4：检索层是怎么设计的

### 30秒版
两阶段设计：粗排用 BM25 和 FAISS 向量检索各取 top-10，RRF 融合排序去重；精排用 CrossEncoder 对融合结果打分取 top-3。BM25 负责精确关键词匹配，向量负责语义理解，两者互补。CrossEncoder 把 query 和 doc 拼在一起做交叉注意力，排序比 bi-encoder 更准。

### 2分钟版（讲故事版，不用背数字）

**第一个问题：单路检索不够用。** BM25 只能匹配关键词，用户搜"文档检索"就搜不到写着"information retrieval"的论文。向量检索有语义理解，但反过来会把"BERT-Large"和"GPT-2"搞混，因为 embedding 太接近了。所以我两路并行，BM25 负责精确匹配，向量负责语义理解，用 RRF 把两路排名融合——一个文档在两路都排前面，融合分数就最高。

**第二个问题：粗排不够准。** 两路各取 top-10 融合后，排序还是比较粗的，因为向量检索用的是 bi-encoder，query 和 doc 分开编码，交互不够。所以我加了 CrossEncoder 精排——它把 query 和 doc 拼在一起输入，内部做交叉注意力，理解更深，从融合结果里精选 top-3。模型很小，本地 CPU 跑，延迟可以忽略。

**效果验证：** 我做了4组对比实验，单 BM25、单向量、混合、混合+精排，每加一层都有明显提升，最终方案的检索指标最好。

> 被追问时才说的细节备忘：
> - Embedding 模型：text-embedding-3-small，1536维
> - FAISS：IndexFlatIP + L2归一化 = 余弦相似度
> - RRF：k=60，原论文推荐值
> - CrossEncoder：ms-marco-MiniLM-L-6-v2，22M参数
> - 数据：MRR 从 0.53→0.62→0.87→1.0
> - 去重 key：source::page::chunk_id

### 记忆锚点
两阶段四步：BM25(10) + Vector(10) → RRF融合 → CrossEncoder精排(3)

## 题5：项目有什么不足，如果继续做会改进什么

### 30秒版
四个方向：一是 FAISS 用暴力搜索，数据量大了要换 HNSW 或 IVFFlat；二是评估体系粗糙，检索用关键词匹配的 Hit Rate，应该用人工标注做 NDCG，生成用同一个 LLM 当 Judge 有 self-bias，应该多模型交叉评估；三是 Agent 的 Plan-and-Execute 不支持动态重规划，后续步骤依赖前面结果时没法填参数；四是 MCP 每次启动新进程开销大，生产环境要换长连接池。

### 2分钟版
四个方向分别展开**怎么改**：

**检索扩展**：IndexFlatIP 是 O(n) 暴力搜索，论文量上十万级别延迟不可接受。换 HNSW（图索引，recall 95%+ 但速度快几十倍）或 IVFFlat（聚类索引，先定位最近的聚类再搜索）。如果数据量继续涨，考虑 Milvus/Qdrant 分布式向量数据库替代单机 FAISS。

**评估体系**：检索评估加人工标注做 NDCG（不只看是否命中，还看排序质量）。生成评估引入 GPT-4 做 Judge 消除 self-bias，或者多个模型交叉评估取平均。增加 A/B 测试框架，在真实用户场景下对比不同方案。

**动态重规划**：当前 Plan-and-Execute 是一次性规划所有步骤。改进是执行完每一步后，把结果传回 Planner 重新规划剩余步骤。比如"搜 RAG 论文然后查第一篇引用"，先搜索拿到论文列表，再规划用第一篇的 paper_id 查引用。

**MCP 长连接**：每次 subprocess 启动约 0.5-1 秒开销。改成启动时建立连接池，复用长连接。可以参考 MCP 协议的 SSE transport 替代 stdio。

### 记忆锚点
四个不足：FAISS扩展、评估体系、动态重规划、MCP长连接
