# 面试30题 — Academic Research Assistant 版本

> 基于项目：学术研究助手（Academic Research Assistant）
> 技术栈：LangGraph + DeepSeek(OpenRouter) + FAISS + CrossEncoder + MCP(ArXiv/Scholar/Notes) + Streamlit + SQLite

---

## 1. 介绍一下这个项目

这个项目是做一个 AI 驱动的学术研究助手，帮助科研人员高效检索、阅读和管理学术论文。系统构建了覆盖**知识问答、工具调用、闲聊**三大场景的多轮对话系统，通过结合 LLM、Hybrid 检索、CRAG（Corrective RAG）和 MCP 工具协议，设计了**三层路由 + RAG子图**的架构。检索层采用 BM25+Vector 的 RRF 混合检索加 CrossEncoder 精排，知识问答层用 CRAG 三级降级策略保证回答的准确性和来源可追溯，工具层接入了 ArXiv、Semantic Scholar、笔记管理三个 MCP Server，并用 Plan-and-Execute + Human-in-the-Loop 的 Agent 架构实现安全可控的工具调用。整体用 LangGraph 编排状态图，支持多轮对话和断点恢复。

---

## 2. 这个项目的难点或挑战是什么？

做这种学术研究助手最大的挑战是**如何保证检索质量和回答的忠实度**。学术论文的内容非常专业，直接用单一检索方式效果不好——BM25 擅长精确关键词匹配但缺乏语义理解，向量检索有语义理解但容易漏掉精确术语。所以我们设计了 **Hybrid 检索 + CrossEncoder 精排**的两阶段架构：第一阶段用 BM25 和 FAISS 向量检索各取 top-10，通过 RRF（Reciprocal Rank Fusion, k=60）融合排序；第二阶段用 CrossEncoder（ms-marco-MiniLM-L-6-v2）对融合结果精排取 top-3。评测结果显示，这个分层设计比单用 BM25 或 Vector 检索的 MRR 和 Hit Rate 都要高很多，最终 MRR 达到 1.0，Hit@3 达到 100%。

---

## 3. 说一下 CRAG（Corrective RAG）是怎么做的

CRAG 是我们项目的核心亮点，解决的是检索结果质量不稳定的问题。传统 RAG 不管检索到的文档质量如何都直接生成，很容易产生幻觉。我们的 CRAG 设计了**三级降级策略**：

检索和精排之后，LLM 会对每篇文档做三元评分（relevant / ambiguous / irrelevant）。如果 ≥50% 文档被评为 relevant，直接用这些文档生成回答；如果评分是 ambiguous（有相关但不够多），就触发**查询改写**，利用已检索到的文档片段辅助改写更精确的检索词，然后重新检索，最多重试 3 次；如果全部 irrelevant 或重试超限，就降级到 **Web Search Fallback**（DuckDuckGo），用搜索结果辅助回答。实测这个链路能把很多原本检索不到的 case 通过改写成功召回。

---

## 4. CRAG 的文档评分具体是怎么做的，讲一下流程

文档评分是 CRAG 的关键环节。具体流程是：精排拿到 top-3 文档后，逐篇截取 content 的前 500 字符（节省 token），连同用户的 query 一起送入 LLM，用专门设计的 Prompt 做三元分类——relevant、ambiguous、irrelevant。

整体评分逻辑是：如果 ≥50% 文档被评为 relevant，整体判定为 relevant，直接进入生成；如果有 relevant 文档但不到 50%，或者有 ambiguous 文档，整体判定为 ambiguous，触发查询改写重试；如果全部是 irrelevant，整体判定为 irrelevant，降级到 Web Search。这个阈值（50%）是我们调测试集调出来的，太高容易触发过多不必要的重试，太低又会放过质量差的文档。

---

## 5. CRAG 如果遇到多轮对话，怎么办

这个问题我们也考虑到了。CRAG 本身处理的是单次检索-评分-生成的链路，但在多轮对话中，用户经常会用"它"、"这篇论文"、"上面说的那个"这种指代表达。所以在 CRAG 子图之前，我们有一个**query_rewrite 节点做指代消解**——它会取最近 6 条对话历史，用 LLM 把模糊的指代改写成完整独立的问题。比如用户先问"Transformer 的自注意力机制是什么"，再问"它的时间复杂度呢"，改写后变成"Transformer 自注意力机制的时间复杂度是多少"。这样 CRAG 收到的永远是语义完整的 query，检索效果就不会受多轮指代的影响。首轮对话没有历史，就直接跳过改写。

---

## 6. 对一些很主观性的 case，模型判别不好，有解决方案吗

在做意图分类的时候确实遇到过这种问题。比如"这篇论文的方法有什么不足"，可以理解为需要检索论文内容回答（knowledge），也可以理解为需要调用 Semantic Scholar 查引用和后续改进工作（task）。还有"帮我整理一下 Transformer 的内容"，是在知识库里检索生成，还是需要做笔记保存？

我们的解决方式是在**意图分类的 Prompt 里加了 8 个 few-shot 示例**，覆盖这些边界 case，明确告诉模型："需要搜索外部论文数据库或保存笔记"→ task，"基于已有知识库内容回答"→ knowledge。另外我们还设置了**安全降级**——如果 LLM 输出的分类结果不在预设的三个类别里（解析失败），默认路由到 chitchat，避免错误路由到工具调用造成不可控的操作。

---

## 7. 如果业务范围发生变更，这个模型如何适配，需要重新训练吗

这个项目的好处是**核心组件不需要重新训练**。意图分类、查询改写、文档评分、回答生成都是用 LLM + Prompt 实现的，不是微调的小模型，所以扩展业务范围只需要改 Prompt 就行。

比如我们想增加一个新的意图类别"论文写作辅助"，只需要在 intent_classify 的 Prompt 里增加这个分类的描述和 few-shot 示例，然后在 LangGraph 的路由里增加对应分支。检索层也是类似——新增论文领域只需要把 PDF 放到 `data/papers/` 目录下，系统启动时会自动加载、切分、构建索引。MCP 工具扩展也很简单，写一个新的 MCP Server，MCPAggregator 启动时会自动发现并注册到路由表里。

唯一需要重新构建的是 **FAISS 向量索引**——新增论文需要重新 Embedding 并更新索引文件，但这个过程是自动的，首次检测到没有缓存就会自动构建。

---

## 8. query 改写模型介绍一下怎么设计的

query 改写在这个项目里有两个层面。

**第一层是主图的指代消解**（query_rewrite_node）：取最近 6 条对话历史，用 DeepSeek + Prompt 把含有指代词的模糊问题改写成完整独立问题。比如"它用了什么 loss function" → "Transformer 模型使用了什么 loss function"。参数是 temperature=0，max_tokens=150，确保改写精确不发散。

**第二层是 CRAG 子图内部的检索改写**（query_rewrite_for_retrieval）：当文档评分为 ambiguous 时触发，这时已经有了初步检索的文档，所以会取前 2 篇文档的中间段落（跳过前30字避免页眉页脚，取接下来300字）作为上下文，辅助 LLM 生成更精确的检索词。temperature=0.3，给一点随机性让改写不至于跟原 query 太雷同。

两层改写的目标不同：第一层解决多轮对话的指代问题，第二层解决检索质量不够时的查询优化问题。

---

## 9. 改写模型最大支持的轮数是多少

指代消解改写取的是最近 **6 条消息**（3 轮对话），CRAG 的检索改写最多**重试 3 次**。

为什么指代消解是 6 条消息？我们测试发现，学术对话场景下用户的指代通常只跨 2-3 轮，超过 3 轮的指代几乎不存在——用户自己也会重新说一个完整的问题。取太多历史反而会引入无关信息干扰改写，增加 token 开销。6 条消息能覆盖绝大多数指代场景，同时保持 Prompt 简洁。

CRAG 重试限制 3 次是因为，如果 3 次改写 + 重新检索都不能召回相关文档，说明知识库里大概率没有这个内容，继续重试只是浪费时间。这时候降级到 Web Search 是更合理的策略。

---

## 10. 如果发生误改写，有没有修复思路

改写确实不是 100% 准确的，我们采用了两层防护。

**第一层是 CRAG 自身的纠错能力**：即使改写后的 query 不够好，CRAG 的文档评分机制会再次判定检索结果的质量。如果改写反而让检索变差了，评分会继续为 ambiguous 或 irrelevant，系统会再次改写或降级到 Web Search，而不是用差的检索结果直接生成幻觉回答。

**第二层是生成阶段的忠实度约束**：generate_node 的 Prompt 严格要求 LLM "只使用上下文中的信息"，temperature 设为 0.1（极低），即使检索结果有偏差，LLM 也不会随意编造内容。如果上下文实在不相关，LLM 会回答"根据现有资料无法确定"，而不是胡说。

未来可以进一步优化的思路是加 RAG 反例库——把历史上的误改写 case 做成向量存起来，改写前先检索是否有类似的反例，如果命中就在 Prompt 里加上反例提示，告诉 LLM 避免这种改写方式。

---

## 11. 改写模型考虑历史的内容了吗，还是只考虑了 QA？

考虑了。指代消解改写不仅看用户的 query 和 LLM 的 answer，还考虑了**整个 AgentState 里的消息流**，包括工具调用的结果。比如用户先让系统"搜索最新的 RAG 论文"，工具返回了论文列表，用户再问"第一篇的作者是谁"——这时候改写需要理解工具返回的论文列表内容，才能正确改写成"XXX 论文的作者是谁"。

我们用 LangGraph 的 `add_messages` Reducer，所有消息（用户输入、LLM 回复、工具返回）都自动追加到 messages 列表里，改写节点取最近 6 条时自然就包含了工具调用的上下文信息。

---

## 12. 为什么不考虑微调训练一个改写模型，而是要用 Prompt 来做？

实际在做这个模块的时候评估过微调方案，最终选择了 LLM + Prompt，主要有三个原因：

**一是数据不好构造**。学术场景的改写非常发散——可能涉及计算机、生物、物理、数学各种领域的论文，指代方式也很多样（"这篇"、"上面那个"、"他的另一篇"等），用有限的人工标注很难覆盖全。而大模型天然有跨领域的语言理解能力，不需要专门的训练数据。

**二是维护成本**。微调模型需要定期更新，一旦对话模式变化或者新增场景就要重新准备数据和训练。Prompt 方案改几行文字就行，迭代速度快得多。

**三是这个任务的实时性要求不高**。改写只在多轮对话时触发，实际我们统计发现学术问答场景中首轮直接提问的占比很高，超过 70%，只有不到 30% 需要改写。用 LLM 多花几百毫秒可以接受，换来的是更高的改写准确率。

---

## 13. 意图分类模型是怎么设计的

意图分类采用的是 **LLM + Prompt + Few-shot** 的方案。我们用 DeepSeek 模型，设计了一个包含 8 个 few-shot 示例的分类 Prompt，把用户的 query 分为三类：

- **knowledge**：需要查询本地知识库/论文内容的问题，比如"Transformer 的自注意力机制是什么"
- **task**：需要调用外部工具的任务，比如"搜索最新的 RAG 论文"、"查一下 Bengio 的 h-index"、"保存这个阅读笔记"
- **chitchat**：问候、感谢、闲聊，比如"你好"、"谢谢"

参数设置 temperature=0、max_tokens=10，确保输出简短确定。分类结果直接用于 LangGraph 的条件路由，把请求分发到 RAG 子图、Agent 工具链或闲聊节点。实测在我们的测试集上准确率很高，主要归功于 few-shot 示例的精心设计——覆盖了各种边界 case。

---

## 14. 仲裁模型考虑多轮对话了吗

考虑了。意图分类的输入不是原始的用户 query，而是经过**指代消解改写**后的 rewritten_query。所以多轮对话中的"搜一下它的引用"，经过改写变成"搜索 Attention Is All You Need 论文的引用"，意图分类就能正确识别为 task（需要调用 Semantic Scholar 的 get_citations 工具），而不是误分为 knowledge。

改写 + 分类的串联设计保证了多轮对话场景下的路由准确性。改写负责解决语义完整性，分类负责解决路由正确性，各司其职。

---

## 15. 如果意图分类路由到 knowledge，但知识库没有相关内容，有没有兜底策略？

有，这就是 CRAG 三级降级策略的价值。路由到 knowledge 后进入 RAG 子图，如果检索结果全部被评为 irrelevant，不会硬生成一个可能有幻觉的回答，而是降级到 **DuckDuckGo Web Search**，用搜索结果辅助回答。Web Search 也失败的话，最终降级到 **LLM 自身知识**，但会在回答中明确标注"这是基于模型自身知识，可能不够准确，建议查阅原始论文"。

为什么不直接让 LLM 凭记忆回答？因为学术场景对准确性要求很高，LLM 的参数化知识可能有过时或错误的内容。所以我们宁可多走一步 Web Search 尝试找到最新信息，实在找不到才用 LLM 兜底，并且明确告知用户答案的来源和可靠性级别。

## 16. 讲一下你的检索层是怎么做的

检索层采用了**两阶段设计：Hybrid 粗排 + CrossEncoder 精排**。

**粗排阶段**用两路并行检索：
- BM25 检索：用 rank-bm25 的 BM25Okapi 算法，分词支持中英文（中文按字切分、英文按空格和标点），取 top-10
- 向量检索：用 text-embedding-3-small 模型生成 Embedding，存入 FAISS（IndexFlatIP + L2 归一化 = 余弦相似度），取 top-10

两路结果用 **RRF（Reciprocal Rank Fusion）** 融合，公式是 `RRF_score = Σ(1/(k+rank_i))`，k=60（标准超参数），去重 key 是 `source::page::chunk_id` 的组合，避免同一文档片段被两路重复召回。

**精排阶段**用 CrossEncoder（`cross-encoder/ms-marco-MiniLM-L-6-v2`），对粗排结果计算 (query, doc) 对的相关性分数，最终保留 top-3。这个模型只有 80MB，在 M 系列 Mac 上 10 条候选文档的精排只要约 0.3 秒。

---

## 17. 为什么检索要做成两阶段设计，有什么考虑

关键考虑是**效果和效率的平衡**。

BM25 擅长精确关键词匹配（比如论文里的专有术语 "BM25Okapi"、"FAISS"），但完全没有语义理解能力——用户问"文档检索"就搜不到只包含"information retrieval"的论文。向量检索有语义理解但容易丢失精确词汇信息——把"BERT-Large"和"GPT-2"的 embedding 可能很接近但实际是不同模型。

**Hybrid 融合**让两者互补：BM25 负责精确召回，Vector 负责语义召回，RRF 融合排序。我们评测了 4 组对比实验——BM25 Only、Vector Only、Hybrid（RRF）、Hybrid + Reranker，结果 Hybrid + Reranker 的 MRR 和 Hit Rate 都是最高的。

**CrossEncoder 精排**是因为 FAISS 用的是 bi-encoder（query 和 doc 分别编码），效率高但交互不够。CrossEncoder 同时看 query 和 doc 的交叉注意力，排序更准，但计算量大。所以先用 Hybrid 粗排缩小到 10-20 条候选，再用 CrossEncoder 精排选出 top-3，这样既保证质量又控制延迟。

---

## 18. BM25 和 Vector 检索的参数量和效果分别怎么样

**BM25**：没有参数量的概念，它是统计方法不是神经网络。用 BM25Okapi 的默认参数（k1=1.5, b=0.75），top-10 召回。速度极快，整个语料库毫秒级返回。缺点是只能做词匹配，没有语义理解。

**Vector 检索**：Embedding 模型是 `text-embedding-3-small`（OpenAI 通过 OpenRouter 调用），向量维度 1536。FAISS 用 IndexFlatIP 做精确最近邻搜索（配合 L2 归一化等价于余弦相似度）。首次构建索引需要调 API 做 Embedding（batch_size=50，有指数退避重试），之后持久化到磁盘，重启直接加载。

**CrossEncoder**：`cross-encoder/ms-marco-MiniLM-L-6-v2`，约 22M 参数（80MB），本地 CPU 推理。max_length=512 token。

**评测结果**（10 个测试用例）：
| 方案 | Hit@3 | Hit@5 | MRR |
|------|-------|-------|-----|
| BM25 Only | 60% | 70% | 0.53 |
| Vector Only | 70% | 80% | 0.62 |
| Hybrid (RRF) | 90% | 90% | 0.87 |
| Hybrid + Reranker | **100%** | **100%** | **1.0** |

每一层叠加都有显著提升：BM25+Vector 混合比单路高 20-30 个百分点，再加 CrossEncoder 精排把最后的 gap 补齐到满分。

---

## 19. RRF 的 k 值是怎么选的，为什么是 60？

k=60 是 RRF 原始论文推荐的标准超参数，在大量 IR 实验中被验证效果稳定。k 的作用是控制排名靠后的文档的权重衰减速度——k 越大，不同排名之间的分数差异越小，融合越"民主"；k 越小，高排名文档的优势越大。

k=60 是 RRF 原始论文（Cormack et al. 2009）推荐的标准值，在 TREC 等大规模 IR 实验中被广泛验证，我们项目的候选量不大（各路 10 条），在这个规模下不同 k 值的差异有限，所以直接采用了业界标准值，没有花时间做 k 的网格搜索。

---

## 20. Hybrid 检索融合时有没有考虑各路的置信度？

目前 RRF 融合是基于排名（rank）而非分数（score）的，这是 RRF 的设计特点——不同检索器的分数分布不一样（BM25 的分数范围和余弦相似度完全不同），直接加权分数需要归一化，而 RRF 用排名就天然避免了这个问题。

不过我们在 BM25 这边做了一个**过滤**：只返回 BM25 分数 > 0 的结果，避免完全不相关的文档参与融合。FAISS 那边因为用的是余弦相似度，所有文档都有分数，通过 top_k 截断控制。

如果要进一步优化，可以考虑给 BM25 和 Vector 设不同的权重系数（比如学术场景下精确术语更重要，可以给 BM25 更高权重），但目前 RRF 等权融合的效果已经很好（MRR=1.0），暂时没有做加权优化。

## 21. 对于 LLM 生成的不标准输出是怎么做后处理的

主要有两类后处理：

**意图分类的输出标准化**：LLM 可能输出 "Knowledge"、"KNOWLEDGE"、"knowledge query" 等各种变体，我们用 `.strip().lower()` 统一转小写，然后检查是否包含在预设的三个类别里。如果不在（比如输出了 "I think this is knowledge"），默认降级到 chitchat，避免路由出错。

**工具规划的 JSON 解析**：AgentPlanner 要求 LLM 输出 JSON 数组 `[{tool, args, purpose}]`，但 LLM 可能输出带 markdown 代码块的 JSON、多余解释文字等。我们先尝试直接 `json.loads()`，如果失败就用正则提取 `[...]` 部分再解析。解析完还会验证每个 step 的 tool 是否在 available_tools 列表里，限制最多 3 步防止幻觉出不存在的工具链。全部解析失败返回空计划，进入 synthesize 告诉用户"无法理解您的需求"，而不是崩溃。

---

## 22. 最后检索和生成效果怎么样

我们做了两个维度的评估：

**检索评估**（evaluate_retrieval.py）：用 10 个覆盖不同学术主题的测试用例（self-attention、RAG、Transformer、BERT、prompt engineering 等），对比 4 种检索方案。最终 Hybrid + Reranker 方案：**Hit@3 = 100%，Hit@5 = 100%，MRR = 1.0**。

**生成评估**（evaluate_generation.py）：用 LLM-as-Judge 做双维度评估：
- **Faithfulness（忠实度）**：判断答案中每个声明是否在检索到的上下文中有依据，输出 0-1 分数 + 无依据声明列表
- **Relevancy（相关性）**：4 级评分标准（1.0 完全回答 / 0.7 部分回答 / 0.4 略有相关 / 0.1 完全不相关）

这个双维度评估比单纯看"答案对不对"更全面——Faithfulness 防止幻觉，Relevancy 确保回答切题。

## 23. 讲一下系统里集成 MCP 怎么做的

MCP（Model Context Protocol）是 Anthropic 提出的开源协议，提供了一种标准化方法让 LLM 连接各种外部数据源和工具。我们在项目里用 MCP 接入了三个外部学术服务。

实现架构是 **MCPAggregator**：启动时扫描所有 MCP Server，每个 Server 通过 stdio 协议（subprocess）通信，Aggregator 自动构建 `tool_name → server_name` 的路由表。当意图分类为 task 时，AgentPlanner 根据 available_tools 列表规划工具调用步骤，ToolExecutor 根据路由表找到对应的 MCP Server 执行调用。

调用流程是：意图分类 → task → AgentPlanner 生成 JSON 计划 → Human-in-the-Loop 确认 → ToolExecutor 逐步执行 → tool_synthesis 综合结果。ToolExecutor 用 ThreadPoolExecutor + asyncio.run() 桥接同步/异步，避免 Streamlit 中嵌套 event loop 的错误，超时设置 30 秒。

## 24. 主要接入了哪些 MCP？

主要接入了三个 MCP Server：

**ArXiv Server**（3 个工具）：
- `search_papers`：搜索论文，返回标题、作者、摘要、arxiv_id、发表日期、PDF 链接
- `get_paper_details`：根据 arxiv_id 获取完整信息，包括分类、DOI、评论
- `generate_bibtex`：生成标准 BibTeX 格式引用，方便直接插入论文

**Semantic Scholar Server**（3 个工具）：
- `find_related_papers`：搜索相关论文，返回标题、作者、年份、引用数、摘要
- `get_citations`：查询引用某篇论文的后续工作
- `get_author_profile`：查作者的 h-index、总引用数、论文数、top 5 高引论文

**Notes Server**（3 个工具）：
- `save_reading_note`：保存阅读笔记到 SQLite（支持标签）
- `search_notes`：全文搜索笔记
- `list_all_notes`：列出最近的笔记

总共 **9 个工具**，覆盖了学术研究中"搜索→分析→记录"的完整工作流。

## 25. 最后的大模型做 NLG 生成是怎么做的，输入是什么

分两种场景：

**知识问答场景**（RAG 子图的 generate_node）：输入是用户 query + 检索到的 top-3 文档。每篇文档格式化为 `[片段N] 来源: xxx, 页码: N\n{content}`，带上来源信息方便溯源。temperature=0.1（极低温度减少幻觉），max_tokens=1000。Prompt 严格要求"只使用上下文中的信息"，不允许编造。

**工具调用场景**（tool_synthesis_node）：输入是用户 query + 所有工具调用的历史记录（包括每步的 tool、args、result、purpose）。LLM 综合这些信息生成结构化回答。temperature=0.3，比 RAG 稍高一点允许更自然的表达。max_tokens=1000。

比如用户问"搜索最新的 RAG 论文"，工具返回了 5 篇论文的元信息，synthesis 节点会把这些整理成结构化列表，包含标题、作者、年份、引用数、摘要摘录等，方便用户快速浏览。

## 26. Web Search Fallback 怎么考虑的，是如何接入系统的

Web Search Fallback 是 CRAG 三级降级的第二级，当本地知识库检索失败（全部 irrelevant 或重试超限）时触发。

方案做了**双重保障**：首先尝试用 `duckduckgo-search` Python 库调 DuckDuckGo API（max_results=3）；如果这个库调用失败（比如 API 限流），备用方案是用 httpx 直接请求 DuckDuckGo Lite 页面，手写 HTML 解析器提取搜索结果。两种方式都失败的话，最终降级到 LLM 自身知识回答，但会明确标注答案来源。

接入方式是在 CRAG 子图里作为一个独立节点，只在 grade_documents_node 评分为 irrelevant 时才路由到这个节点。搜索结果会格式化后传给 LLM 生成回答，Prompt 里会说明"以下是网络搜索结果"，让 LLM 区分来源。

---

## 27. Web Search 一般会比本地检索慢，这块是怎么优化的

首先，Web Search 不是每次都触发的——只有在 CRAG 评分为 irrelevant 时才走这条路径。实际我们发现，如果本地知识库的论文覆盖比较全，大部分问题都能在本地检索阶段解决，只有少量超出知识库范围的问题才需要 Web Search。

另外，我们在 CRAG 的重试机制里做了优化：ambiguous 状态最多重试 3 次改写+重新检索，如果第一次检索就是全部 irrelevant（而不是 ambiguous），会直接跳过重试直接降级到 Web Search，省去不必要的重试时间。

DuckDuckGo 搜索本身也做了限制——max_results=3，不贪多，只取最相关的 3 条结果就够 LLM 生成回答了。

## 28. 系统最后有没有做评测，是如何做的指标评测

系统做了**模块化评测 + 端到端评测**。

**模块化评测**：
- **检索评估**（evaluate_retrieval.py）：10 个测试用例，4 组对比实验（BM25 / Vector / Hybrid / Hybrid+Reranker），指标是 Hit@3、Hit@5、MRR。Hybrid+Reranker 最终 MRR=1.0，Hit@3=100%
- **生成评估**（evaluate_generation.py）：LLM-as-Judge 双维度评估——Faithfulness（忠实度）检查幻觉，Relevancy（相关性）检查切题程度。3 个端到端测试问题，汇报平均分

**端到端评测**：
- 三条路径验证：闲聊→正确路由到 chitchat_node ✅，知识问答→走 RAG 子图 ✅，工具调用→走 Agent 链 ✅
- CRAG 链路验证：构造了"首次检索不到→改写→重新检索成功"的 case，确认整个降级链路工作正常

**单元测试**（test_core.py）：5 个 TestCase 覆盖 Document 创建、TextSplitter 切分、HybridRetriever 去重 key 一致性、意图路由逻辑、AgentState 字段完整性。

## 29. 模型服务有没有做压测，性能怎么样？

本地模型（BM25 + CrossEncoder）因为是 CPU 推理，主要关注延迟而非 QPS：
- **BM25 检索**：毫秒级，全语料库扫描非常快
- **CrossEncoder 精排**：M 系列 Mac 上 10 条候选约 0.3 秒
- **FAISS 向量检索**：IndexFlatIP 精确搜索，小规模数据集（几百条 chunk）毫秒级

LLM 调用走 OpenRouter API，延迟取决于 DeepSeek 的 API 响应速度，一般首 token 在 1-2 秒。整个 RAG 链路端到端（检索+精排+评分+生成）大约 3-5 秒。工具调用链路（规划+执行+综合）因为有多次 LLM 调用和外部 API 调用，大约 5-10 秒。

系统用 SQLite Checkpointer 做持久化，多轮对话的状态恢复是磁盘 IO 级别（<100ms）。FAISS 索引有持久化缓存，第二次启动跳过 Embedding 构建，启动速度从几十秒降到几秒。

## 30. 做完这个项目最大的收获是什么

最大的收获是理解了**RAG 系统从"能跑"到"好用"的差距有多大**。

一个朴素的 RAG（检索+生成）搭起来很快，但实际效果很差——检索不到就幻觉，检索到了但不相关也硬生成。加了 CRAG 之后，系统学会了"自我纠错"：检索结果不好就改写重试，实在不行就降级到 Web Search，最后还降级不了就诚实说"不知道"。这种**多级降级 + 忠实度约束**的设计让系统从"有时候对"变成了"要么对，要么诚实说不知道"，大幅减少了幻觉。

另一个收获是 **LangGraph 的状态图编排真的很适合复杂 Agent 系统**。用普通代码写 if-else 分支、重试循环、Human-in-the-Loop 中断恢复，代码会变得非常混乱。LangGraph 把每个步骤抽象成节点，把路由逻辑抽象成条件边，加上 SQLite Checkpointer 支持断点恢复，整个系统的可维护性和可扩展性都好很多。新增一个功能就是加一个节点和一条边的事。

最后是**二八原则**：评估数据显示 Hybrid+Reranker 在标准测试集上已经是 MRR=1.0 了，但实际使用中总会有长尾的 bad case。CRAG 的改写重试 + Web Search 降级就是用 20% 的额外复杂度覆盖了那些 80% 的基础方案搞不定的尾部问题。工程上不应该追求 100% 完美，而是在合理复杂度内尽可能提高覆盖率。

## 31. 你的 Prompt 是怎么设计的，有什么技巧？

项目里有 8 个 Prompt，设计上有几个原则：

**一是控制输出格式**。意图分类的 Prompt 最后一句是"分类结果："，用冒号引导 LLM 直接输出标签而不是一段分析。CRAG 评分也一样，"评估结果："引导输出 relevant/ambiguous/irrelevant 其中一个词。

**二是 few-shot 示例**。意图分类放了 8 个示例覆盖边界 case，比如"把这段摘要翻译成中文"是 task 不是 knowledge，"论文里提到的实验结果是什么"是 knowledge 不是 task。这些边界示例比写规则描述更有效，LLM 从例子中学习模式比理解抽象规则更准确。

**三是不同任务用不同 temperature**。分类/评分用 0（确定性），闲聊用 0.7（自然多样），RAG 生成用 0.1（低幻觉），改写用 0.3（微量随机避免跟原 query 完全一样）。

**四是 RAG 生成的忠实度约束**。Prompt 里明确写了"只使用提供的上下文中的信息"和"如果上下文信息不足以回答，明确告知用户"，这两条是防幻觉的关键。

---

## 32. chunk_size 为什么是 512，chunk_overlap 为什么是 50？

chunk_size=512 是一个平衡点。太小（比如 128），每个 chunk 信息量太少，检索到了也不够回答问题，而且 chunk 数量暴增导致检索变慢。太大（比如 2048），一个 chunk 里塞了多个主题，检索相关性会下降——用户问 self-attention，检索到的 chunk 里可能一半在讲 attention 一半在讲 position encoding，噪音太多。512 字符在学术论文场景下大约是 1-2 个段落，基本能覆盖一个完整的知识点。

chunk_overlap=50 是为了防止**语义断裂**。如果一个关键句子恰好被切在两个 chunk 的边界上，两个 chunk 都只有半句话，检索就可能漏掉。50 字符的重叠保证边界附近的信息在相邻 chunk 里都有。50 大约是 chunk_size 的 10%，这个比例在实践中效果稳定——重叠太多浪费存储和计算，太少起不到作用。

---

## 33. 为什么选 text-embedding-3-small 而不是其他 Embedding 模型？

几个考虑：

**一是效果够用**。text-embedding-3-small 在 MTEB benchmark 上表现优秀，1536 维，对学术论文的语义理解足够好。我们评测 Hybrid+Reranker 达到 MRR=1.0，说明 Embedding 质量没有成为瓶颈。

**二是成本低**。通过 OpenRouter 调用，价格远低于 text-embedding-3-large（3072 维）。我们的数据量不大（几百个 chunk），用更贵的模型提升有限。

**三是有 CrossEncoder 兜底**。Embedding 模型只是粗排的一半（另一半是 BM25），后面还有 CrossEncoder 精排。所以 Embedding 不需要特别精准，只需要在 top-10 里把相关文档召回来就行，精准排序交给 CrossEncoder。

如果要优化，可以考虑换成开源的 BGE-M3 或 GTE 模型在本地跑，省掉 API 调用费用和网络延迟，但目前这个规模下没必要。

## 34. 这个项目有什么不足，如果继续做你会改进什么？

**检索层**：当前 FAISS 用的是 IndexFlatIP（暴力精确搜索），数据量小没问题，但如果论文量上到十万级别，需要换成 IndexIVFFlat 或 HNSW 做近似最近邻，牺牲一点精度换取速度。

**评估体系**：当前检索评估是基于关键词匹配的 Hit Rate，比较粗糙。更好的做法是用人工标注的 relevance judgment 做 NDCG 评估，但标注成本高。生成评估用的是 LLM-as-Judge，存在 LLM 自己评自己的偏差（self-bias），可以引入多个不同的 Judge 模型交叉验证。

**Agent 链路**：Plan-and-Execute 模式下 step2 的参数不能依赖 step1 的结果。比如"搜索 RAG 论文，然后查第一篇的引用"，规划时不知道第一篇是什么，没法填 get_citations 的参数。要解决这个需要加 dynamic replanning——执行完 step1 后重新规划后续步骤。

**多用户/并发**：当前 MCP 每次调用都启动一个新的 subprocess，开销大。如果上生产环境需要换成长连接池复用。

---

## 35. LLM-as-Judge 具体怎么做的，有什么局限？

两个维度：

**Faithfulness（忠实度）**：把检索到的上下文和 LLM 生成的回答一起给 Judge 模型，让它检查回答里的每个声明是否在上下文中有依据。输出 JSON 格式：`{score: 0-1, unsupported_claims: [...]}`。score 越高说明幻觉越少。

**Relevancy（相关性）**：把用户问题和生成的回答给 Judge 模型，4 级评分（1.0 完全回答 / 0.7 大部分 / 0.4 部分相关 / 0.1 没回答）。

**局限性**：最大的问题是 **Judge 和 Generator 是同一个模型（DeepSeek）**。LLM 评价自己生成的内容会有系统性偏高的倾向（self-bias）。理想情况下应该用不同的模型做 Judge，比如用 GPT-4 评估 DeepSeek 的输出，或者用多个模型交叉评估取平均。另一个局限是 LLM-as-Judge 的评分不完全稳定，同一个 case 多次评分可能有小幅波动，所以我们把 temperature 设为 0 来尽量减少这种波动。

## 36. FAISS 的 IndexFlatIP 是什么意思？为什么做了 normalize_L2？

```python
faiss.normalize_L2(embeddings)
self.index = faiss.IndexFlatIP(dim)
```

IndexFlatIP 是 FAISS 的**内积（Inner Product）暴力搜索**索引。但我们想要的是**余弦相似度**。数学上：**两个单位向量的内积 = 余弦相似度**。所以先用 `normalize_L2` 把所有向量归一化为单位长度，再用内积搜索，效果等价于余弦相似度搜索。query 进来时也做了同样的归一化（`_embed_query` 里也调了 `normalize_L2`）。

**追问：为什么不直接用 IndexFlatL2（欧氏距离）？** 因为欧氏距离受向量长度影响，两个语义相似但 embedding 幅度不同的向量，欧氏距离可能很大。余弦相似度只看方向不看长度，对 Embedding 更合适。

---

## 37. RRF 融合那段代码具体怎么算的？

```python
for rank, (doc, _) in enumerate(bm25_results):
    rrf_scores[doc_id] = rrf_scores.get(doc_id, 0) + 1 / (k + rank + 1)
```

每个文档在每路检索里有一个排名（rank，从 0 开始）。RRF 给每个排名一个分数 `1/(60+排名+1)`。排名第 0 的文档得分 `1/61 ≈ 0.0164`，排名第 9 的得分 `1/70 ≈ 0.0143`。如果一个文档同时被 BM25 和 Vector 两路召回，两个分数**累加**，所以两路都排在前面的文档 RRF 总分最高。

`rank+1` 是因为 rank 从 0 开始，但 RRF 公式定义排名从 1 开始。

## 38. TextSplitter 的 overlap 具体是怎么实现的？

```python
overlap_text = current_chunk[-self.chunk_overlap:] if current_chunk else ""
current_chunk = overlap_text + "\n" + para if overlap_text else para
```

当一个段落放不进当前 chunk 了，先把当前 chunk 存起来，然后取当前 chunk 的**最后 50 个字符**作为 overlap 拼到下一个 chunk 的开头。相邻两个 chunk 之间有 50 字符的重叠区域，避免关键信息被切断。

对于超长段落（单段超过 512 字符），`_force_split` 用滑窗切：每次取 512 字符，起始位置只前进 `512-50=462` 字符，保证相邻窗口有 50 字符重叠。

## 39. BM25 的分词为什么这么写？

```python
tokens = re.findall(r"[\u4e00-\u9fff]|[a-zA-Z0-9]+", text.lower())
```

**中英文混合分词**：`[\u4e00-\u9fff]` 匹配单个中文字符（按字切分），`[a-zA-Z0-9]+` 匹配连续英文/数字（按词切分）。`text.lower()` 统一转小写避免大小写影响匹配。

**追问：中文按字切分效果好吗？** 理想情况应该用 jieba 分词，但学术论文大部分是英文，中文只是少量元数据，按字切分够用。BM25 只是粗排的一路，后面有 CrossEncoder 精排兜底。

---

## 40. Embedding 的指数退避重试具体怎么做的？

```python
for retry in range(3):
    try:
        resp = self.client.embeddings.create(...)
        break
    except Exception as e:
        wait = 2 ** retry  # 第1次等1s，第2次等2s
        time.sleep(wait)
```

最多重试 3 次。第 1 次失败等 1 秒（2^0），第 2 次失败等 2 秒（2^1），第 3 次还失败就抛异常。这叫**指数退避（exponential backoff）**，API 失败通常是限流，间隔越来越长给服务端喘息时间，比固定间隔重试更合理。

---

## 41. CrossEncoder 的 predict 输入是什么格式？

```python
pairs = [(query, doc.content) for doc in docs]
scores = self.model.predict(pairs)
```

输入是 **(query, document) 的文本对列表**。CrossEncoder 对每一对打一个相关性分数。跟 Bi-Encoder 不同——Bi-Encoder 分别编码后算距离，CrossEncoder 把两段文本**拼在一起**输入模型，内部 attention 让 query 和 document 的每个 token 做交叉注意力，理解更深，排序更准。

## 42. interrupt() 返回的是什么？resume 怎么传回来的？

```python
user_decision = interrupt({"type": "human_confirm", "message": "...", "plan": plan})
```

`interrupt()` **暂停图执行**，把传入的 dict 存到 State。外部 `chat()` 函数检测到 `state.next` 不为空就知道图被暂停了，把信息返回给用户。用户确认后调 `app.invoke(Command(resume="yes"))`，这个 `"yes"` 成为 `interrupt()` 的**返回值**赋给 `user_decision`，图从暂停处继续执行。

---

## 43. BM25 为什么过滤 scores > 0？

```python
results = [(self.documents[i], float(scores[i])) for i in top_indices if scores[i] > 0]
```

BM25 对完全没有关键词匹配的文档打分为 0。不过滤的话，top-10 里可能有很多 0 分文档参与 RRF 融合，拉低整体质量。过滤掉 0 分 = **跟 query 连一个词都不重合的文档没有召回价值**。
