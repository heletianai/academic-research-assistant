## 题6：意图分类是怎么设计的（原题13+6+14 合并）

### 30秒版
用 LLM + Prompt + 8个 few-shot 示例，把用户 query 分成三类：knowledge（查知识库）、task（调工具）、chitchat（闲聊）。输入是经过指代消解改写后的完整 query，不是原始输入，所以多轮对话也能正确分类。temperature=0 保证确定性，解析失败默认降级到 chitchat，不会误触工具调用。

### 2分钟版

**为什么用 LLM+Few-shot 而不是训练分类器？** 三个类别，边界 case 很多——"帮我整理 Transformer 内容"到底是检索回答还是保存笔记？靠规则写不完，靠标注数据训练成本高还不灵活。LLM 天然理解语义，8 个 few-shot 示例覆盖边界 case 就够了，新增意图只要加一行示例。

**多轮怎么处理？** 分类的输入不是用户原话，而是改写后的 rewritten_query。"搜一下它的引用"先被改写成"搜索 Attention Is All You Need 的引用"，分类自然就是 task。改写管语义完整性，分类管路由正确性，各司其职。

**容错怎么做？** LLM 可能输出"Knowledge"、"KNOWLEDGE"、"I think this is knowledge"各种格式。统一 strip().lower()，不在三个类别里就降级到 chitchat。宁可当闲聊处理，也不误触工具调用。

### 记忆锚点
三类（knowledge/task/chitchat）+ 输入是改写后的 query + 解析失败降级 chitchat

## 题7：MCP 和 Agent 工具调用是怎么做的（原题23+24+21 合并）

### 30秒版
接了三个 MCP Server：ArXiv、Semantic Scholar、Notes，共9个工具。意图分类为 task 后，AgentPlanner 用 LLM 生成 JSON 执行计划，Human-in-the-Loop 让用户确认后才执行，ToolExecutor 逐步调用对应的 MCP Server，最后 synthesis 节点综合结果生成回答。

### 2分钟版

**架构怎么设计的？** MCPAggregator 启动时扫描所有 MCP Server，每个 Server 通过 stdio 子进程通信，自动构建 tool_name → server_name 的路由表。AgentPlanner 看到 available_tools 列表后生成计划，不需要知道工具在哪个 Server 上，路由表自动分发。

**安全怎么保证？** 三层防护：一是 JSON 解析后校验每个 tool 是否在 available_tools 里，防止 LLM 幻觉出不存在的工具；二是限制最多3步，防无限调用；三是 Human-in-the-Loop，用户确认后才执行。LangGraph 的 interrupt() 暂停图执行，用户确认后 Command(resume="yes") 恢复。

**LLM 输出不标准怎么办？** 先试 json.loads()，失败就用正则提取 JSON 数组部分再解析。全部失败返回空计划，告诉用户"无法理解需求"，不会崩溃。

### 记忆锚点
3个Server/9个工具 → Planner生成计划 → 人工确认 → Executor执行 → Synthesis综合

## 题8：Prompt 是怎么设计的（原题31）

### 30秒版
项目里有8个 Prompt，四个设计原则：一是控制输出格式，用"分类结果："引导 LLM 直接输出标签；二是 few-shot 示例覆盖边界 case；三是不同任务不同 temperature，分类用0、生成用0.1、闲聊用0.7、改写用0.3；四是 RAG 生成严格限制"只用上下文信息"防幻觉。

### 2分钟版

**格式控制**：意图分类 Prompt 最后一句是"分类结果："，CRAG 评分是"评估结果："，用冒号引导 LLM 直接输出一个词。不这么做的话 LLM 会输出一大段分析再给结论，浪费 token 还难解析。

**few-shot 的价值**：意图分类放了8个示例，关键是边界 case——"把摘要翻译成中文"是 task 不是 knowledge，"论文里的实验结果是什么"是 knowledge 不是 task。这些边界用规则写不清楚，但给 LLM 看两个例子它就懂了。

**temperature 策略**：需要确定性输出的（分类/评分）用0，需要忠实于文档的（RAG 生成）用0.1，需要一点随机性的（改写）用0.3避免跟原 query 太像，需要自然多样的（闲聊）用0.7。每个值都有明确理由。

**防幻觉**：RAG 生成 Prompt 写了两条硬约束——"只使用提供的上下文中的信息"和"如果信息不足，明确告知用户"。这两条加上 temperature=0.1，是整个系统防幻觉的最后一道防线。

### 记忆锚点
四原则：格式引导 + few-shot边界case + temperature分级 + 忠实度约束

## 题9：评测是怎么做的（原题22+28+35 合并）

### 30秒版
两个维度：检索评估用 Hit@K 和 MRR，10个测试用例4组对比实验，验证每层叠加都有提升。生成评估用 LLM-as-Judge 评两个维度——Faithfulness 检查幻觉、Relevancy 检查切题。还做了端到端的三条路径验证和 CRAG 降级链路验证。

### 2分钟版

**检索怎么评的？** 10个覆盖不同学术主题的测试用例，对比4种方案。关键不是最终分数，而是**每层叠加都有显著提升**——这证明了架构设计的每一层都在发挥作用，不是堆复杂度。样本量10个不大，但趋势很明确。

**生成怎么评的？** 用 LLM-as-Judge 做双维度：Faithfulness 把上下文和回答一起给 Judge，检查每个声明是否有依据，输出分数+无依据声明列表；Relevancy 把问题和回答给 Judge，4级评分。两个维度分开评比单看"对不对"更全面——答案可能对但包含编造的细节（Faithfulness 低），也可能不编造但没回答问题（Relevancy 低）。

**局限是什么？** Judge 和 Generator 是同一个 DeepSeek，存在 self-bias。如果要改进，应该用不同模型交叉评估，或者引入人工评测做校准。

### 记忆锚点
检索：Hit@K+MRR+4组对比 | 生成：Faithfulness+Relevancy 双维度 | 局限：self-bias

## 题10：做完这个项目最大的收获（原题30）

### 30秒版
最大收获是理解了 RAG 从"能跑"到"好用"的差距。朴素 RAG 搭起来很快但幻觉严重，加了 CRAG 后系统学会自我纠错——检索不好就改写，改写不行就 Web Search，实在不行就说"不知道"。从"有时候对"变成"要么对，要么诚实说不知道"。

### 2分钟版

**第一个收获：多级降级的工程价值**。CRAG 不是一个算法，是一种工程思维——任何环节都可能失败，所以每条路都要有兜底。检索不好→改写重试，重试不行→Web Search，Web Search 也不行→LLM 兜底但标注来源。这种思维在做任何系统都适用。

**第二个收获：LangGraph 适合复杂 Agent**。用普通代码写分支、重试、Human-in-the-Loop 中断恢复，代码会非常乱。LangGraph 把步骤抽象成节点、路由抽象成条件边，加 SQLite Checkpointer 支持断点恢复，可维护性好很多。新增功能就是加一个节点一条边。

**第三个收获：二八原则**。基础 RAG 解决 80% 的问题，CRAG 用 20% 额外复杂度覆盖剩下的长尾 bad case。工程上不追求 100% 完美，而是在合理复杂度内最大化覆盖率。

### 记忆锚点
三个收获：多级降级思维 + LangGraph编排价值 + 二八原则
